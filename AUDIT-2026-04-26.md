## Problèmes réels classés par priorité

### 🔴 Phase 1 — Urgent, low risk

#### 1. Pas de concurrency global entre daily-generate et weekly-generate

**Fichiers** : `.github/workflows/daily-generate.yml` (lignes 50-52), `.github/workflows/weekly-generate.yml` (lignes 41-43)

**Problème** : Chaque workflow a son propre concurrency group. Si un daily et un weekly sont dispatchés manuellement en même temps, ils peuvent écrire simultanément à `data/state/episode_history_{en,fr}.json` et `docs/` → race condition sur git push.

**Fix** :
```yaml
# Dans daily-generate.yml ET weekly-generate.yml
concurrency:
  group: generate-drafts-${{ github.ref }}
  cancel-in-progress: false
```

---

#### 2. settings._apply_locale() : pas de validation URL vs model ID

**Fichier** : `src/ai_press_review/settings.py` lignes 289-321

**Problème** : Si `LLM_FALLBACK_MODEL_EN` contient une URL (ex. `https://api.anthropic.com/v1`) au lieu d'un model ID, le workflow preflight le détecte mais le code Python lui-même ne valide pas. Tout appel hors workflow (script local, dry-run) crashe silencieusement.

**Fix** :
```python
# Après résolution du model per-locale
if per_locale_editor and per_locale_editor.startswith(('http://', 'https://')):
    raise ValueError(f"LLM_EDITOR_MODEL_{suffix} doit être un model ID, pas une URL : {per_locale_editor}")
```

---

#### 3. _normalize_sections_payload : logs insuffisants

**Fichier** : `src/ai_press_review/editorial/generator.py` lignes 815-818

**Problème** : Le log actuel dit "rewrapping N keys" sans lister lesquelles. En cas de partial-return du LLM, impossible de savoir quelles clés ont été rewrappées et lesquelles manquent.

**Fix** :
```python
logger.info(
    "LLM returned section keys at JSON root instead of under 'sections' — rewrapping (%d keys: %s)",
    len(flat),
    ", ".join(flat.keys()),
)
```

### 🟡 Phase 2 — Maintenabilité

#### 4. ~95% de code dupliqué entre daily-generate.yml et weekly-generate.yml

**Problème** : Le preflight (850+ lignes), la matrice locale, les retry loops, les git push patterns sont identiques dans les deux fichiers. Un fix doit être appliqué deux fois. Drift accidentel probable.

**Différences réelles** :
- `profile` default : `daily` vs `weekly_recap`
- `timeout-minutes` : 25 vs 30
- `LLM_TIMEOUT_SECONDS` : absent vs 360

**Approche recommandée (B — reusable workflow)** :

Créer `.github/workflows/generate-draft-impl.yml` (reusable workflow) :
```yaml
on:
  workflow_call:
    inputs:
      profile:
        type: string
      timeout_minutes:
        type: number
      llm_timeout_seconds:
        type: number
```

`daily-generate.yml` et `weekly-generate.yml` l'appellent via `uses:` avec leurs paramètres propres.

**Avantage vs fusion complète** : Garde deux points d'entrée distincts (logs GitHub lisibles, crons séparés, debugging facile).

## Architecture globale — Points forts à préserver

### EN → FR manifest sharing (déjà en place)
```bash
# daily-generate.yml + weekly-generate.yml
if [ "$APR_LOCALE" = "fr" ]; then
  EN_MANIFEST="data/sources/en/${RUN_DATE}.json"
  if [ -f "$EN_MANIFEST" ]; then
    SOURCES_ARG="--sources-file $EN_MANIFEST"
  fi
fi
```
FR reuse le manifest crawlé par EN → pas de double crawl. LLM FR tourne indépendamment avec `prompt_weekly_fr.txt`.
